## 侦察

### 阶段 1 — 寻找攻击面

我寻找该域名的唯一侦察方法只是一个简单的 Google Dork：

```
site:*.REDACTED.com -www

```

通过这个简单的 dork，我获取了大量子域名。但其中有一个特别引起了我的注意，它看起来像一个 IP 地址，但点号被替换为 —（短横线）：10-10-10-16.mobile.REDACTED.com访问该域名时，它是一个路由器登录门户，似乎是公司内部使用的。

我尝试了所有默认凭证，但都无法登录。于是我进入第二阶段侦察。

### 阶段 2 — 客户端源码分析

我快速运行了 Javascript Bookmarklet 来获取客户端代码中的所有端点，并写到页面上。它获取了当前子域名下的大量端点。 我尝试了它给我的所有端点，但很多都只是 401 forbidden 页面。一个端点返回了 200 OK，并且页面很有趣。/guest/preview.cgi暴露了一个管理员页面，允许我们修改任何页面内容，例如条款与条件页、登录页等。

### 阶段 3 — 发现有趣功能

它有一个叫Edit mode的切换按钮，打开后可以编辑选定页面的元素。我们可以修改文本、按钮、图片等。

为了测试，我尝试在条款与条件页面的“同意”按钮上添加This was modified by 1-day。点击“保存”后，我不确定是否会保存。真相时刻到了，我点击并等待……

BOOM！页面提示修改已保存，我刷新页面……

这告诉我们，我们可以在完全未认证的情况下永久修改页面内容。这本身就是严重的配置错误。但我不是那种只会报告无用漏洞的人…

## 利用

有了修改页面内容的能力，我立刻想到了我最不喜欢的漏洞类型——跨站脚本（XSS）。

### 深入漏洞

保存编辑后的页面时，它会发送一个 POST 请求到/guest/portal_admin_upload.cgi，请求体类型为multipart/form-data，要编辑的页面元素在edit_page字段中，格式为 JSON。 我捕获请求后发现，“我同意”按钮对应的 key 是agree_button，其值会直接显示在页面上。例如：

```
{"agree_button":"This is an Agree Button"}

```

于是我在 Burpsuite 中更新请求，加入了一个基础的 XSS payload：

```
{"agree_button":"<script>alert('PoC by 1-day')</script>"}

```

发送请求后，返回 200 OK 并显示“保存成功”。

Payload 保存成功 刷新页面后：

XSS payload 被执行 所以我们得到一个未认证的存储型 XSS 漏洞。 但等等……我不是那种只报告无用漏洞的人！

## 链接到潜在账户接管

### 计划 A

当我们得到存储型 XSS 并希望升级为账户接管时，第一直觉是抓取用户 Cookie 并发送到攻击者服务器。 但这个应用的 Cookie 设置非常严格，有 HttpOnly 标志，并且 Path 只针对主页面，因此无法获取管理员 Cookie。 计划 A 失败。

### 计划 B（制作完美利用链）

这里展示一个巧妙的利用链，可以通过存储型 XSS 潜在获取管理员凭证。 由于我们有未认证的存储型 XSS，我们可以存储一个 payload，将访问者重定向到攻击者服务器，但需要隐蔽且有目的。 我使用HTTrack克隆了整个路由器登录页面到本地，获取所有 JS、HTML、CSS、CGI 文件，并保持目录结构。

```
httrack https://10-10-10-16.mobile.REDACTED.com/

```

10 分钟后，我完美克隆了登录页面。 然后，我修改cgi-bin/index.html以将凭证发送到我的服务器。

#### 第 1 步

然后修改cgi-bin/index.html将凭证发送到我的服务器。 为了隐蔽，还会在 3 秒后重定向到预览页（未被 XSS 感染的页面）。通过参数portal_id可以控制编辑不同页面，如：

- preview.cgi?portal_id=1为编辑页
- preview.cgi?portal_id=2为未感染页

流程：

- 受害者访问 https://10-10-10-16.mobile.REDACTED.com/guest/preview.cgi
- 静默重定向到攻击者服务器（克隆登录页）
- 受害者输入凭证登录
- 页面显示“登录成功”，3 秒后重定向到preview.cgi?portal_id=2（正常页）
- 凭证记录在攻击者服务器上

成功展示了未认证存储型 XSS 的实际影响。

## 影响

利用该链，攻击者可以创建一个高度可信的登录页面，管理员不知情地访问可能导致账户接管。 若攻击者获取内部路由器管理权限，可以修改 DNS/proxy 指向恶意服务器，窃取网络内所有设备信息，并可能升级为极其严重的攻击。

## 结论

我本可以像大多数猎人一样直接报告 XSS……但我不是那种人。公司不需要弹窗显示“被黑了”，他们需要真正会造成损失的安全问题。最终，攻击的呈现方式才是关键。 本文详细记录了我发现该漏洞的过程。希望大家喜欢这篇文章。
