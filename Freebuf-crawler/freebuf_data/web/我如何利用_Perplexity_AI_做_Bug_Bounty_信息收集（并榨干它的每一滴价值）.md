## 我如何利用 Perplexity AI 做 Bug Bounty 信息收集（并榨干它的每一滴价值）

当大多数人谈到信息收集（Recon）时，他们想到的是运行几个工具，导出成千上万个子域名，然后……迷失在噪音中。
我以前也是这样。
后来我开始使用 Perplexity——不是把它当作某种神奇的"帮我找漏洞"按钮，而是把它当作一个会思考的研究助手。一旦我学会了如何给它喂对提示（prompt），我的信息收集方式完全变了。
这篇文章不是理论，而是我在接触一个新项目时的实际工作流，目标是在不触碰任何超出范围（out-of-scope）内容的情况下，绘制出它的全部攻击面。

### 第 0 步：设定基本规则

在我输入目标域名到 Perplexity 之前，我会先设定好场景。如果你不告诉它你在做什么，它只会给你一些通用的垃圾信息。
这是我每次信息收集会话都会粘贴的**"基准"提示**：

```
Context（上下文）：我有权限在以下范围内的资产上执行信息收集。Goal（目标）：仅使用合法、被动的 OSINT，识别并优先排序攻击面。Scope（范围）：[在此粘贴项目提供的域名/IP/通配符]Deliverable（产出）：必要时用表格、项目符号、JSON。标明来源。标记未知项。Constraints（限制）：除非明确允许，否则不得扫描。仅限被动情报。

```

这样可以让 Perplexity 聚焦在我的目标上，也强迫我自己明确我要找什么。

### 第 1 步：真正理解范围

很多人只扫一眼赏金计划说明，抓取通配符域名就开始狂扫。这样很容易碰到超范围的资产，最后报告被拒。
我的做法是把政策全文丢进 Perplexity，然后让它拆解成：

- 纯资产列表
- 根域名、FQDN、CIDR
- 包含类型、范围说明和限制的 JSON

Perplexity 经常能帮我找到那些你可能会忽略的细节，比如政策中间夹着的一个单独 IP 段。

### 第 2 步：子域名枚举策略

不要直接导出 5 万个子域名淹死在数据里，而是让 Perplexity 帮我制定一个三阶段的被动枚举计划：

- 来源（CT 日志、DNS 历史、分析 ID）
- 示例查询
- 可靠性评分
- 安全验证方法

它经常会提醒我一些忘了存在的 CT 日志网站，或者分析 ID 的关联枚举技巧，能带我找到全新的子域名。

### 第 3 步：ASN 和基础设施映射

这里能找到真正隐藏的资产——那些不在 Cloudflare 背后的。

```
使用公共 WHOIS、IPinfo、Shodan、Censys 查询这些域名的 ASN 和 IP 段。表格格式：ASN → 网段 → 托管商 → 自有还是第三方。

```

一旦拿到 ASN，就能发现一些规律，比如生产环境旁边还藏着个裸 IP 的测试环境。

### 第 4 步：CT 日志 = 被遗忘的宝藏

我的一个常用技巧：

```
从这些域名获取历史 CT 日志数据，列出：- FQDN- 首次出现时间- 最后出现时间- 组织名不匹配情况

```

老的记录往往能指向那些已经"退役"但依然存活的系统。

### 第 5 步：聪明的 Google Dorking

**不要随便乱用site:搜索，让 Perplexity 生成针对目标定制的 Dork：**

```
为该目标生成 Google/Bing/Brave 搜索语句，寻找：- 开发/测试环境- 暴露的目录- GitHub 提及- 公开文档（PDF/DOCX/PPTX）

```

PDF 和 DOCX 的元数据里经常藏着邮箱或内网主机名。

### 第 6 步：第三方情报

供应商和 SaaS 工具会扩大攻击面。

```
根据分析 ID、追踪像素、客服插件找出这些域名关联的第三方服务。表格格式：服务 → 证据 → 关联子域名。

```

然后用这些分析 ID 去反查，看它们是否用在公司其他没列入范围的域名上。

### 第 7 步：公共代码提及

我现在不手动搜 GitHub 了。

```
查找提及这些域名或员工邮箱的公共仓库。表格：仓库 → 所有者 → 信息收集价值。

```

这里不是为了找明文泄漏，而是找内部工具、API 端点、旧提交里的遗忘子域名。

### 第 8 步：归档信息收集

Wayback Machine 里有大量被遗忘的端点。

```
按年份列出这些域名的唯一 URL。标出已消失的管理后台、API 文档、登录页面。

```

然后检查这些端点是否换到别的子域名上继续存在。

### 第 9 步：DNS 配置错误

无需发一个数据包也能发现劫持机会。

```
检查这些域名是否有失效的 CNAME、陈旧的 MX、未使用的 TXT。表格：记录 → 发现 → 安全验证方法。

```

我找到的很多"低垂果实"都是 DNS 里挖出来的。

### 第 10 步：像专业人士一样优先排序

信息收集没有优先级就是一团乱。

```
给每个资产按 1–5 分评分：- 潜在漏洞可能性- 一旦被攻破的业务影响

```

这样我就有一个短名单来做后续主动测试。

### 第 11 步：设定时间限制

Perplexity 能帮你把信息收集做成时间冲刺计划：

```
根据我的收集结果，制定一个 60 分钟和一个 4 小时的被动信息收集冲刺计划。

```

适合想快速出结果和做深度挖掘的不同日子。

### 第 12 步：随时保持报告就绪

我现在边做边格式化：

```
格式：资产 | 证据来源 | 发现 | 优先级 | 安全后续

```

这样等要提交漏洞报告时，直接复制就行。

### 我常用的三个"榨干"提示

想让 Perplexity 超越基础，这三个很好用：

```
1. 我还有哪些信息收集角度没试过？2. 哪些发现和第三方服务有关？3. 哪些历史变化暗示着新部署？

```

### 最后的想法

信息收集是你能悄悄领先其他猎人的阶段。每个人都能跑相同的工具，但不是每个人都能连起这些点。
Perplexity 不是替你思考，而是放大你的思考。当你让它和你一起思考，而不是替你思考时，你就能画出别人没有的地图。在漏洞赏金狩猎里，这张地图就是一切。
