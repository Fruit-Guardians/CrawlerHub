### 特权容器

以提升的权限运行容器，攻击者可能借此控制底层主机系统。例如：以 root 级别访问权限和无限制能力运行容器。

不安全代码中，容器使用--privileged标志启动，开启特权模式，赋予容器对主机系统的无限制访问，可能破坏安全边界。

```
# 不安全：特权容器

FROM ubuntu
...
# 以特权模式运行容器
RUN docker run -it --privileged ubuntu /bin/Dockerfile

```

安全代码通过不启用特权模式运行容器，限制容器对系统资源的访问，降低权限提升和未授权访问主机的风险。

```
# 安全：非特权容器

FROM ubuntu
...
# 不启用特权模式运行容器
RUN docker run -it ubuntu /bin/Dockerfile

```

### 暴露的容器 API

未对容器 API 进行适当认证或访问控制，导致攻击者可操控或窃取敏感信息。例如：暴露 Docker API 无任何认证或加密。

不安全代码中，容器 API 在 8080 端口暴露，且无认证授权机制，允许无限制访问，易受攻击。

```
# 不安全：暴露容器 API 无认证/授权

FROM nginx
...
# 在 8080 端口暴露容器 API
EXPOSE 8080

```

安全代码通过内部暴露 API 并结合反向代理或 API 网关进行认证授权，增强安全性。

```
# 安全：通过认证/授权保护容器 API

FROM nginx
...
# 内部暴露容器 API 端口 8080
EXPOSE 8080

# 使用反向代理或 API 网关进行认证/授权
server {
listen 443 ssl;
server_name api.example.com;

ssl_certificate /etc/nginx/certs/server.crt;
ssl_certificate_key /etc/nginx/certs/server.key;
ssl_client_certificate /etc/nginx/certs/ca.crt;
ssl_verify_client on;

location / {
proxy_pass http://localhost:8080;
}
}
此配置要求客户端提供有效证书，未认证无法访问API

```

安全最佳实践包括：

- 实施强认证和授权机制，如 OAuth、JWT，并基于角色权限控制访问。
- 使用 TLS 加密保护客户端与容器 API 通信。
- 定期监控和记录 API 活动，检测异常。
- 应用速率限制和请求节流，防止滥用和拒绝服务攻击。

**不安全容器配置:**

```
# 不安全：未使用安全隔离运行容器

require 'docker'

# 使用默认设置创建容器
container = Docker::Container.create('Image' => 'nginx')
container.start

```

安全代码通过 HostConfig 参数增强安全隔离，禁用特权模式、移除所有能力、阻止权限提升。

```
# 安全：增强安全隔离运行容器

require 'docker'

# 创建带安全设置的容器
container = Docker::Container.create(
  'Image' => 'nginx',
  'HostConfig' => {
    'Privileged' => false,           # 禁用特权模式
    'CapDrop' => ['ALL'],            # 移除所有能力
    'SecurityOpt' => ['no-new-privileges']  # 阻止权限提升
  }
)
container.start

```

### 容器镜像篡改

修改或替换容器镜像为恶意版本，可能包含恶意软件、后门或漏洞组件。例如：注入恶意代码窃取敏感信息。

以下代码直接拉取并运行镜像，未验证完整性，易受镜像篡改攻击。

#### 容器镜像篡改案例与利用步骤

1.获取合法镜像攻击者先从官方或目标仓库拉取合法镜像作为基础，例如：

```
docker pull nginx:stable

```

2.基于合法镜像构建恶意镜像

利用镜像分层特性，攻击者在合法镜像基础上添加恶意程序（如反向Shell、木马等），构建新的恶意镜像：

```
FROM nginx:stable
RUN apt-get update && apt-get install -y curl
ADD backdoor.sh /usr/local/bin/backdoor.sh
RUN chmod +x /usr/local/bin/backdoor.sh
CMD ["/usr/local/bin/backdoor.sh"]

```

或直接在容器中修改后commit成新镜像。

3.重命名恶意镜像为原镜像名称和标签为了掩盖篡改，攻击者将恶意镜像打上与原镜像相同的名字和标签：

```
docker tag malicious-image nginx:stable

```

4.推送恶意镜像到目标镜像仓库攻击者将恶意镜像推送回被攻破的镜像仓库，覆盖原镜像：

```
docker push nginx:stable

```

```
# 未验证完整性拉取并运行镜像

require 'docker'

# 拉取镜像
image = Docker::Image.create('fromImage' => 'nginx')

# 运行镜像
container = Docker::Container.create('Image' => image.id)
container.start

```

安全代码通过计算并比对镜像的 SHA256 摘要进行完整性验证，防止篡改。

```
# 安全：拉取并运行镜像时验证完整性

require 'docker'
require 'digest'

image_name = 'nginx'
image_tag = 'latest'

image = Docker::Image.create('fromImage' => "#{image_name}:#{image_tag}")

expected_digest = Digest::SHA256.hexdigest(image.connection.get("/images/#{image.id}/json").body)
actual_digest = image.info['RepoDigests'].first.split('@').last
if expected_digest != actual_digest
  raise "镜像完整性验证失败: #{image_name}:#{image_tag}"
end

container = Docker::Container.create('Image' => image.id)
container.start

```

### 不安全的容器配置

容器配置错误，如弱访问控制或过度权限，导致容器或环境被攻破。例如：运行带不必要能力或不安全挂载点的容器。

不安全代码使用默认设置创建容器，可能存在安全隐患。

```
# 不安全：不安全配置运行容器

require 'docker'

container = Docker::Container.create('Image' => 'nginx')
container.start

```

安全代码通过 HostConfig 应用安全配置，如只读文件系统、移除所有能力、禁止权限提升、使用桥接网络隔离、端口绑定限制访问。

```
# 安全：安全配置运行容器

require 'docker'

container = Docker::Container.create(
  'Image' => 'nginx',
  'HostConfig' => {
    'ReadOnly' => true,               # 文件系统只读
    'CapDrop' => ['ALL'],             # 移除所有能力
    'SecurityOpt' => ['no-new-privileges'],  # 禁止权限提升
    'NetworkMode' => 'bridge',        # 桥接网络隔离
    'PortBindings' => { '80/tcp' => [{ 'HostPort' => '8080' }] }  # 端口绑定限制
  }
)
container.start

```

### 拒绝服务（DoS）

通过过载容器资源或利用运行时漏洞，破坏容器化应用可用性。例如：通过大量请求发起 DoS 攻击。

不安全 Dockerfile 未限制资源，容器可无限制消耗资源，易受 DoS 攻击。

```
# 不安全：无限资源分配的 Dockerfile

FROM nginx:latest

COPY app /usr/share/nginx/html

EXPOSE 80

CMD ["nginx", "-g", "daemon off;"]

```

安全示例强调应根据需求配置资源限制，如 CPU、内存，通常通过容器编排平台或 Docker-compose 实现。

```
version: '3'
services:
  nginx:
    image: nginx:latest
    ports:
      - 80:80
    volumes:
      - ./app:/usr/share/nginx/html
    deploy:
      resources:
        limits:
          cpus: '0.5'
          memory: '256M'

```

### 内核漏洞

利用内核或主机操作系统漏洞，获得未授权访问或控制容器。例如：利用内核漏洞提升权限，攻破容器。

不安全代码忽略内核漏洞，直接运行容器。

```
# 不安全：忽略内核漏洞

docker run -d ubuntu:latest /bin/Dockerfile

```

安全代码示例使用 kubehunter 工具扫描内核漏洞，发现漏洞后采取修复措施，再创建容器。

```
# 安全：检查内核漏洞

kubehunter scan

# 根据扫描结果修复漏洞

docker run -d ubuntu:latest /bin/Dockerfile

```

### 共享内核利用

多个容器共享同一内核，内核漏洞可被利用影响多个容器。例如：利用内核漏洞访问同一主机上的多个容器。

不安全代码安装易受攻击的软件包并运行易受攻击应用，存在容器逃逸风险。

```
# 不安全：易受容器逃逸攻击

FROM ubuntu:latest

RUN apt-get update && apt-get install -y vulnerable-package

CMD ["vulnerable-app"]

```

安全代码只包含必要且安全的软件包，定期更新并打补丁，运行安全应用，降低逃逸风险。

```
# 安全：缓解容器逃逸风险

FROM ubuntu:latest

RUN apt-get update && apt-get upgrade -y && apt-get install -y secure-package

CMD ["secure-app"]

```

此外，采用容器隔离技术、使用安全增强内核、监控日志有助于提升安全。

### 不安全的容器编排

容器编排平台（如 Kubernetes）配置错误或漏洞，导致未授权访问、权限提升或敏感信息泄露。例如：利用配置错误的 Kubernetes 集群获取敏感资源。

不安全代码中 Pod 启用特权模式，容器拥有提升权限，攻击者可利用该权限攻击主机或其他容器。

```
# 不安全：易受权限提升攻击

apiVersion: v1
kind: Pod
metadata:
  name: vulnerable-pod
spec:
  containers:
    - name: vulnerable-container
      image: vulnerable-image
      securityContext:
        privileged: true  # 启用特权模式

```

安全代码禁用特权模式，减少潜在攻击面。

```
# 安全：缓解权限提升风险

apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  containers:
    - name: secure-container
      image: secure-image
      securityContext:
        privileged: false  # 禁用特权模式

```

同时应配置 RBAC 策略、网络隔离、及时打补丁并监控异常。

### 导出所有 Secret

导出 Kubernetes 集群中所有 Secret 指未授权提取存储的敏感信息，可能导致进一步攻破。

不安全代码利用kubectl get secrets --all-namespaces导出所有 Secret，假设拥有访问权限。

```
# 不安全：导出所有 Secret

kubectl get secrets --all-namespaces -o json > secrets.json

```

安全建议包括：

- 实施最小权限原则，严格权限分配和审计。
- 配置基于角色的访问控制（RBAC）。
- 利用命名空间隔离资源。
- 对 Secret 进行加密存储和传输。
- 监控 API 审计日志。
- 定期轮换 Secret。
- 强化集群访问安全，如多因素认证。

### 窃取 Pod 服务账号令牌

窃取 Pod 服务账号令牌指未授权获取 Kubernetes 集群中运行 Pod 的服务账号令牌，令牌可访问 Kubernetes API 及资源。

不安全代码通过kubectl exec命令读取 Pod 内令牌文件。

```
# 不安全：窃取服务账号令牌

kubectl exec <pod-name> -- cat /var/run/secrets/kubernetes.io/serviceaccount/token

```

安全建议：

- 限制 Pod 权限，遵循最小权限原则。
- 使用 RBAC 限制权限。
- 避免挂载服务账号令牌。
- 定期轮换令牌。
- 监控 Pod 活动日志。
- 使用 Pod 安全策略限制权限提升。

### 创建管理员 ClusterRole

创建管理员 ClusterRole 指在 Kubernetes 中创建具备管理员权限的 ClusterRole，绑定服务账号，实现集群管理。

不安全代码创建权限过宽的管理员角色，绑定服务账号，存在安全风险。

```
# 不安全：创建管理员 ClusterRole

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: admin-clusterrole
rules:
- apiGroups: [""]
  resources: ["*"]
  verbs: ["*"]

apiVersion: v1
kind: ServiceAccount
metadata:
  name: admin-serviceaccount
  namespace: kube-system

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: admin-clusterrolebinding
subjects:
- kind: ServiceAccount
  name: admin-serviceaccount
  namespace: kube-system
roleRef:
  kind: ClusterRole
  name: admin-clusterrole
  apiGroup: rbac.authorization.k8s.io

```

安全代码遵循最小权限原则，仅赋予必要权限管理指定资源。

```
# 安全：创建有限权限管理员 ClusterRole

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: admin-clusterrole
rules:
- apiGroups: [""]
  resources: ["pods", "deployments"]
  verbs: ["get", "list", "create", "update", "delete"]

apiVersion: v1
kind: ServiceAccount
metadata:
  name: admin-serviceaccount
  namespace: kube-system

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: admin-clusterrolebinding
subjects:
- kind: ServiceAccount
  name: admin-serviceaccount
  namespace: kube-system
roleRef:
  kind: ClusterRole
  name: admin-clusterrole
  apiGroup: rbac.authorization.k8s.io

```

### 创建客户端证书凭证

创建客户端证书凭证指为 Kubernetes 集群中的特权用户生成客户端证书，用于认证和访问集群。

不安全代码仅生成私钥和证书请求，未完成证书签发流程。

```
# 不安全：生成私钥和证书请求

openssl genrsa -out client.key 2048

openssl req -new -key client.key -out client.csr -subj "/CN=client"

cat client.csr

```

安全代码使用 Kubernetes CSR API，完成证书请求创建、批准及证书获取。

```
# 安全：创建客户端证书凭证

apiVersion: certificates.k8s.io/v1beta1
kind: CertificateSigningRequest
metadata:
  name: client-csr
spec:
  groups:
  - system:authenticated
  request: (base64-encoded CSR)
  usages:
  - client auth

kubectl certificate approve client-csr

kubectl get csr client-csr -o jsonpath='{.status.certificate}' | base64 -d > client.crt

echo "客户端证书:"
cat client.crt

echo "客户端私钥:"
openssl rsa -in client.key -text

```

### 创建长期有效令牌

创建长期有效令牌指为 Kubernetes 服务账号生成长期有效的访问令牌，攻击者可利用其持久访问集群。

不安全代码创建服务账号令牌但未指定过期时间，默认令牌有效期。

```
# 不安全：创建服务账号令牌

kubectl create serviceaccount long-lived-token-sa

kubectl get secret $(kubectl get serviceaccount long-lived-token-sa -o jsonpath='{.secrets[^0].name}') -o jsonpath='{.data.token}' | base64 -d

```

安全代码通过定义 TokenRequest 对象，指定令牌过期时间。

```
# 安全：创建长期有效令牌

apiVersion: authentication.k8s.io/v1
kind: TokenRequest
metadata:
  name: long-lived-token
spec:
  audience: api
  expirationSeconds: 2592000  # 30 天，可调整
  tokenRequest:
    metadata:
      name: serviceaccount-name
      namespace: namespace-name

kubectl create -f token-request.yaml

kubectl get secret $(kubectl get tokenrequest long-lived-token -o jsonpath='{.status.secretName}') -o jsonpath='{.data.token}' | base64 -d

```

### 通过 hostPath 卷挂载进行容器突破

攻击者通过 hostPath 卷挂载整个节点根文件系统，逃逸容器环境，访问主机敏感文件或执行特权操作。

不安全代码示例挂载主机根目录，读取主机/etc/passwd文件。

```
apiVersion: v1
kind: Pod
metadata:
  name: hostpath-container-breakout
spec:
  containers:
  - name: attacker-container
    image: busybox
    command: ["/bin/sh", "-c"]
    args: ["cat /host/etc/passwd"]
    volumeMounts:
    - name: hostpath-volume
      mountPath: /host
  volumes:
  - name: hostpath-volume
    hostPath:
      path: /

```

安全代码通过限制权限和禁止特权提升防止突破。

```
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  containers:
  - name: trusted-container
    image: busybox
    command: ["/bin/sh", "-c"]
    args: ["echo 'Access denied'"]
    securityContext:
      allowPrivilegeEscalation: false

```

### 通过 node/proxy 权限进行权限提升

利用 Kubernetes 节点代理 API 权限绕过审计和控制，提升为集群管理员权限。

不安全代码创建具备nodes/proxy资源全部权限的 ClusterRole 并绑定服务账号。

```
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: nodes-proxy-role
rules:
- apiGroups: [""]
  resources: ["nodes/proxy"]
  verbs: ["*"]
---
apiVersion: rbac.authorization.k8s.io/v1
```

### 运行特权 Pod

在 Kubernetes 中运行特权 Pod 等同于在节点上以 root 权限运行，存在权限提升风险。

不安全代码创建启用特权模式的 Pod。

```
apiVersion: v1
kind: Pod
metadata:
  name: privileged-pod
  namespace: your-namespace
spec:
  containers:
  - name: privileged-container
    image: busybox:latest
    command: ["sleep", "3600"]
    securityContext:
      privileged: true

```

安全代码避免启用特权模式，降低安全风险。

```
apiVersion: v1
kind: Pod
metadata:
  name: non-privileged-pod
  namespace: your-namespace
spec:
  containers:
  - name: non-privileged-container
    image: busybox:latest
    command: ["sleep", "3600"]

```

通过不设置或设置privileged为 false，Pod 以普通用户权限运行，减少权限提升风险。
